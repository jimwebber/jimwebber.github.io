Making Metadata Work Harder – SSDL and Crash Recovery

Posted on August 8, 2006 by jim Posted in Uncategorized — No Comments ↓

Update: fixed dodgy formatting

A few days back, Savas blogged about the Newcastle University team that entered the IEEE Service-Oriented Computing Contest with a cool service-oriented system designed to automate real estate transactions. One of the interesting  experimental aspects of the system was the major actors involved were described using SSDL and the  team created  the world’s first SSDL engine to run the business protocols that comprised the solution.

While I was happy just to marvel at this super piece work, it took an email exchange between Philipp Reinecke and Patric Fornasier on the team’s mailing list to slap me back into actually thinking critically about it, and about the profound effect that metadata has on dependable computing systems in particular. The SSDL engine that the team implemented validates a business protocol by recording (some of) the details of messages that it sends and receives in a persistent store. The individual messages exchanges are themselves stateless, therefore making the service scalable (to the limits of its underlying datastore) and recoverable. What is interesting is not so much that a validation of an entire conversation is possible (of course this isn’t possible with WSDL), but that it is in fact possible to leverage SSDL for more than that.

It’s actually not a huge intuitive leap to think about SSDL as a DSL for Web Services. If that DSL had appropriate framework support then the kind of validation engine that the team built by hand could be automatically generated (much like how Rails automatically generates database schemas and bindings for Ruby Web apps). However there is no need to stop simply at automatically generating the protocol validation framework for a business process. As messages are sent and received, the underlying framework stores them for validation purposes, but it is equally possible to put them to use for another (generally horrible) task – crash recovery.

Rewind to February when I last caught up with Martin. He gave an interesting talk about a pattern he’s observed which he’s calling “Chain of Events” where an application records all the stimuli it receives from the outside world. This pattern makes crash recovery straightforward (assuming internally deterministic behaviour), since all that needs to happen is for those external stimuli to be replayed back to the application at recovery time.

In the case of a Web Service described with SSDL, it is possible to have a framework which builds on top of the protocol validation code to create crash recovery code. Since crash recovery code is a horrible affair which is prone to having bugs crop up at inconvenient times (i.e. at recovery time!), any level of automation around this will be a significant boost to enterprise Web Services development. All of which is, I think, presents some pretty exciting opportunities. No longer is metadata only for automatically generating client-side stubs, but can we put metadata to automate (for example) the creation of transactional scopes for specific subsets of related activities (where the relations are expressed, of course, as part of the SSDL description)? Ditto for security aspects.  Indeed I’m intrigued by the notion of applying policies across a finely targeted set of related interactions (which is not  possible with today’s technology), and using tooling to automate the resulting secure, reliable, transacted business protocols that emerge.

Is this viable, or am I completely barking up the wrong tree?
