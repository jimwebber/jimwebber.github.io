My First C# 3.0 Bug…

Posted on November 15, 2007 by jim Posted in Uncategorized — No Comments ↓

I’m with Erik Meijer in his Ã˜redev workshop on Linq. We’re going through the motions with C# 3.0 at the moment, and I’ve found my first bug, or at least something which Erik believes is inconsistent with the language specification.

Consider the following equivalent anonymous types:

var myPerson = new { FirstName = “Jim”, LastName = “Webber” };
var yourPerson = new { FirstName = “Savas”, LastName = “Parastatidis” };

Since these types are the same under the covers, it’s OK for me to make the following assignment:

myPerson = yourPerson;

So far so sensible, but what happens if I change the structure of the type of yourPerson in some hopefully insignificant way:

var yourPerson = new { LastName = “Parastatidis”, FirstName = “Savas” };

To my mind (and Erik’s and the C# 3.0 spec), I should still be able to make the same assignment because the underlying anonymous type is still structurally the same (just a couple of string properties):

myPerson = yourPerson;

Except right now the compiler doesn’t like it and complains that it can’t convert one anonymous type to the other. So as of Beta 2 of Visual Studio 2008 this is broken, but I guess you folks with the RTM version might know better…

Update: Erik’s had chance to check the C# language spec and says the observed behaviour is consistent that spec, so it’s not a bug at all, just an eccentricity. He also says that in some circumstances it turns out that ordering is important enough to warrant this behaviour. I’m guessing those circumstances are to do with C# internals rather than anything I’ll do as a user, but interesting nonetheless!
