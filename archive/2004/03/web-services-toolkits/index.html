Web Services Toolkits

Posted on March 4, 2004 by jim Posted in Uncategorized — No Comments ↓

I’m coding again, which is a joy. Better still I am working on the .Net
platform which I used to look at with envy while working in the piecemeal
world of Java Web Services for the last few years. I don’t have anything bad
to say about the individual components in the Java space (for instance Axis
and FX rocks), but there are a lot of components from a lot of vendors out
there that you end up trying to link together (why do we need different
infoset representations for SAAJ and
DOM?). Being from a single vendor, everything in .Net just seems to work
together harmoniously with everything else.

The service I’m currently working on is to support the long lived naming of
logical resources in Grid applications. It is basically a store for GRM
Documents (see WS-GAF for an
overview) which allow resources like scientific instruments and datasets to
be named, and possibly adorned with metadata.

In theory, this is a trivial service to implement since it accepts GRM
Documents (which are XML) and stores them on unique key (known as a GRI).
There’s some reasonably interesting GRM Document storage and crash recovery
code, but essentially the service is simple.

However, the GRM Document uses substitution groups (which are a useful bit
of kit from XML Schema), and while xsd.exe deals with this reasonably well
(unlike JAXB), I couldn’t seem to get the ASP.Net serialisation stuff to do
what I want (and still generate the original WSDL).

So I’ve upgraded to WSE, which initially seems weird because where ASP.Net
gives you all kinds of groovy attributes to do all the hard work and does
its best to give you application-level objects from XML, WSE is focussed on
lower-level messaging so it initially looks like a downgrade. WSE puts you
right down at the SOAP message level and leaves you to figure out what to do
with incoming and outgoing messages.

This raises the bar a bit for a developer since you now have to understand
SOAP, the SOAP processing model, and XML programming in order get anywhere
(whereas ASP.Net hid all that behind application-level objects and
serialisation attributes). However the flexibility it offers is great. As
the service developer I generally know what messages look like and how to
handle them – after all I designed the service. Although it might look more
“primitive” than ASP.Net, the model is far more natural and encourages a
message-oriented way of thinking about service development.
