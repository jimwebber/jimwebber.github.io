NAnt

Posted on March 30, 2004 by jim Posted in Uncategorized — No Comments ↓

As I mentioned previously, for my current
work I’ve been able to switch from Java to the .Net framework and I’m pretty
happy with the switch – with the advent of
WSE, .Net is the platform for Web Services development. Although the
tool support for .Net is great, I was troubled by the fact I couldn’t script
more advanced functions like service deployments, copying, packaging and so
on easily.

When I was working in the Java space, one tool really made my life
easier, Apache Ant. With Ant I scripted
all manner of Herculean tasks like deploying a “hello world” service on
Oracle 9i (actually I deployed transaction coordination services, but even
“hello world” was tough enough). Once written I could purge the
details out of my head knowing that all I needed was captured as Ant tasks.

Now that I’m working with
Savas
and the services we’re writing are hosted on the opposite side of the planet
to me, the VS.Net build and deploy has become insufficient. I wrote some Ant
scripts to automate the building and publishing of our services, and even
went as far as to make simple callouts to the command line C# compiler and
NDoc. But Ant and .Net are not really
great bedfellows since all the good stuff in Ant is aimed at Java
development. I’ve made the switch to NAnt which, as its name suggests, is
like Ant but focussed on automating common .Net development tasks. With a
few minor tweaks, and the replacement of messy command line invocations with
“proper” tasks my build system is now working natively in .Net. I might even
remove the various JDKs from my box at this rate.

There are a couple of things that I’d like to see though (typing as if
this blog could grant wishes). The first is that I’d like the next iteration
of Visual Studio.Net to support the NAnt scripting language
natively so that I can do more complex things than just compile/build within
the same environment. In fact it would be useful to be able to
debug those scripts in the same environment too. Secondly I’d like to see
Steve Loughran write a new book that
covers NAnt, based on his current book,
“Java Development with Ant.”

Update: 30/03/2004 14:17 Sydney, Australia

Time to port all the unit tests to
NUnit.
Thankfully we’re lax on the QA
front, the benefit of being research- rather than product-oriented! I am
giving serious consideration to dropping the older versions of the JDK
from my box.

Update: 13/04/2004 14:09 Sydney, Australia

I am not the only one who would like to see tighter integration between
